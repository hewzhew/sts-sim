# Mechanics High-Risk Backlog

This is the short list to keep visible while card/relic/monster parity work
continues. These are simulator-mechanics risks, not policy risks.

## 1. Draw Pile And CardGroup Ordering

Risk:

- Java `CardGroup` top-of-deck semantics do not map directly to ad hoc Rust
  `Vec` operations.
- Mixing `push`, `insert(0)`, and direct zone mutation can silently reverse top
  and bottom behavior.

Rule:

- All draw-pile movement must go through a single zone API.
- Direct `push` / `insert(0)` on draw-pile-like zones should be treated as a
  review finding unless it is inside the zone API itself.

Must keep testing:

- top placement;
- bottom placement;
- random insertion;
- draw order after generated cards and status cards.

## 2. Generated Cards Entering Piles

Risk:

- Java generated-card actions distinguish hand, discard, draw-pile top,
  draw-pile bottom, and random draw-pile insertion.
- `addToRandomSpot` has its own semantics and must not accidentally make a
  generated card the next draw when Java would not.

Examples to keep revisiting:

- `MakeTempCardInDrawPileAction`
- `MakeRandomCardInDrawPile`
- `Nilry's Codex`
- `Mark of Pain`
- `Wild Strike`
- Corrupt Heart status-card insertion

Rule:

- Generated-card placement must be represented as an explicit destination, not
  inferred from a helper name.

## 3. Random Target Selection

Risk:

- Java random-target actions usually choose the target when the action executes,
  not when the card is played.
- `MonsterGroup.getRandomMonster(..., aliveOnly=true)` excludes dead and
  half-dead monsters in ways that affect multi-hit and post-lethal behavior.

Examples:

- `DamageRandomEnemyAction`
- `AttackDamageRandomEnemyAction`
- `BouncingFlaskAction`
- relic and power damage-to-random-enemy hooks

Rule:

- If Java target selection is deferred to action execution, Rust must also
  defer it.

## 4. Choice And Suspend Actions

Risk:

- Grid/hand/draw/discard/exhaust choices are not interchangeable.
- Duplicate cards require stable instance refs.
- `min`, `max`, `any_number`, `can_cancel`, selected-so-far, and ordered
  choices affect both legality and replay.

Examples:

- `PutOnDeckAction`
- `Headbutt` / `Hologram`-style choices
- `Setup`
- `SecretTechnique`
- `SecretWeapon`
- `Exhume`
- `Warcry`
- scry / order-card flows

Rule:

- Choice actions must carry source zone, selection constraints, reason, and
  stable card refs. Do not encode them as generic "pick a card" actions.

## 5. Post-Combat Cleanup

Risk:

- Java `GameActionManager.clearPostCombatActions()` keeps and drops different
  action families after lethal damage.
- Damage actions generated by random-target actions may be retained differently
  from the original random-target wrapper.

Examples:

- `PummelDamageAction`
- `BouncingFlaskAction`
- `DamageRandomEnemyAction`
- vampire/reaper heal timing

Rule:

- Lethal damage handlers must run Java-like cleanup at the same boundary as
  Java. Do not treat post-combat cleanup as a generic queue clear.

## 6. Card Instance Copy And Mutation

Risk:

- Java uses concrete card instances, UUIDs, `misc`, cost-for-turn, cost-for-combat,
  and `GetAllInBattleInstances`-style mutation in ways that affect both the
  current combat and master-deck metadata.

Examples:

- `RitualDagger`
- `GeneticAlgorithm`
- `Rampage`
- `GlassKnife`
- cost modifiers from `Corruption`, `Enlightenment`, `Madness`, Snecko, and
  generated-card actions

Rule:

- Copies, generated cards, and "all matching instances" mutation must preserve
  instance identity rules explicitly. Do not rely on `CardId` equality where
  Java uses a concrete card or UUID.

## Adjacent Watchlist

These are not part of the six-item core list, but they should remain visible:

- X-cost cards and `energy_on_use` / `free_to_play_once` timing;
- Chemical X timing;
- powers/relics that mutate card costs while actions are already queued;
- combat-end hooks versus post-combat reward RNG;
- public observation/replay leakage once AI collection starts again.
